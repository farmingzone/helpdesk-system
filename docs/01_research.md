# 01. 리서치: 티켓 시스템 업무 구조와 데이터 흐름 분석

## 1) 내부 Helpdesk의 본질
내부 Helpdesk는 "고객 응대"보다 "사내 운영 효율"이 핵심이다.  
즉, 핵심 질문은 다음과 같다.
- 어떤 요청이 들어왔는가?
- 지금 누가 처리하고 있으며 어디까지 진행됐는가?
- 처리 결과가 추적 가능한가?
- 운영 관점에서 병목이 어디에서 발생하는가?

MVP에서는 이 중에서도 **요청 추적성과 처리시간 가시성**에 집중한다.

## 2) 업무 구조 (Business Workflow)

### 2.1 참여 주체
- 요청자: 업무 요청/민원 생성 주체
- 처리자: 요청을 분류/진행/완료하는 운영 담당
- 시스템: 상태 이력 기록, 통계 집계, 조회 제공

### 2.2 상태 모델 (MVP)
- `접수`: 요청이 등록되었고 아직 본격 처리 전
- `처리중`: 담당자가 실제 해결 작업을 수행 중
- `완료`: 요청 해결이 끝나고 종료된 상태

핵심은 상태값 그 자체보다 "상태 전이 이벤트"다.  
전이가 발생할 때마다 로그가 남아야 감사 가능성과 통계 정확도가 확보된다.

### 2.3 업무 단계별 세부 동작
1. 요청 등록
- 요청자가 제목/내용 등을 입력해 티켓 생성
- 시스템은 `ticket` 레코드 생성 + 초기 상태 `접수` 저장
- 동시에 `ticket_history`에 생성 이벤트 기록

2. 처리 시작
- 처리자가 상태를 `처리중`으로 변경
- 시스템은 티켓 현재 상태 업데이트
- 변경 전/후 상태, 처리자, 메모를 이력으로 저장

3. 처리 완료
- 처리자가 상태를 `완료`로 변경
- 시스템은 완료 시각(`resolved_at`)을 기록
- 이력 로그 추가 후, 통계 집계 대상에 편입

## 3) 데이터 흐름 (Data Flow) 상세 분석

## 3.1 엔티티 간 관계
- `ticket` (현재 스냅샷): "지금 상태"를 빠르게 조회하기 위한 기준 테이블
- `ticket_history` (이벤트 로그): 상태 변경/메모 기록의 시간축 데이터

관계:
- `ticket` 1 : N `ticket_history`
- 실무적으로 UI 목록은 `ticket` 중심, 상세 타임라인은 `ticket_history` 중심

## 3.2 생성/갱신 흐름
1. 생성 시점
- 입력 검증 후 `ticket` INSERT
- 같은 트랜잭션에서 `ticket_history` INSERT(이벤트 타입: CREATED)

2. 상태 변경 시점
- 전이 가능 여부 검증(`접수`->`처리중`, `처리중`->`완료`)
- `ticket.status` UPDATE
- 완료 전이 시 `ticket.resolved_at` 기록
- `ticket_history` INSERT(이벤트 타입: STATUS_CHANGED)

3. 통계 조회 시점
- 기준: 완료된 티켓(`resolved_at IS NOT NULL`)
- 처리시간 = `resolved_at - created_at`
- 평균 처리시간 = 완료건 처리시간의 평균

## 3.3 정합성 포인트
- 티켓 상태 변경과 이력 기록은 하나의 원자적 단위로 처리되어야 함
- `완료` 상태인데 `resolved_at`이 비어 있으면 안 됨
- 이력이 있는데 현재 상태와 불일치하면 안 됨(최신 이력 == 현재 상태)

## 3.4 조회 패턴과 인덱스 관점
- 목록 조회: 상태별 최신 티켓 목록
  - 인덱스 후보: `(status, created_at DESC)`
- 상세 조회: 특정 티켓의 이력 타임라인
  - 인덱스 후보: `(ticket_id, created_at ASC)`
- 통계 조회: 완료 티켓 평균 처리시간
  - 인덱스 후보: `(resolved_at)` 또는 `(status, resolved_at)`

MVP에서는 과최적화보다 단순하고 일관된 쿼리 구조를 우선한다.

## 4) 운영 시나리오 기반 분석

### 시나리오 A: 단건 요청 처리
1. 요청자: "사내 VPN 접속 오류" 티켓 등록
2. 시스템: `접수` 상태로 저장 + 생성 로그
3. 처리자: 조사 시작 시 `처리중` 전이 + 메모
4. 처리자: 해결 후 `완료` 전이 + 해결 메모
5. 통계: 평균 처리시간 계산에 해당 건 반영

관찰 포인트:
- 진행 단계가 명확해 팀 내 인수인계가 쉬워짐
- 이력 기반으로 "언제 지연됐는지" 추적 가능

### 시나리오 B: 처리 지연 점검
1. 처리중 상태 티켓의 체류 시간이 길어짐
2. 이력 로그를 통해 마지막 업데이트 시각 확인
3. 업무 배분 조정 또는 우선순위 재정렬

MVP는 자동 알림이 없어도, 이력/상태만으로 운영 판단의 기초를 제공한다.

## 5) MVP에서 의도적으로 단순화한 요소
- 우선순위(SLA) 모델 미도입
- 담당자 자동 할당 미도입
- 첨부파일/코멘트 스레드 미도입
- 다단계 승인 흐름 미도입

단순화의 이유:
- 포트폴리오 1차 목적은 "핵심 플로우 완결성" 검증
- 범위를 줄여야 오히려 데이터 정합성과 도메인 표현이 선명해짐

## 6) 실무 관점 트레이드오프
- 현재 상태 스냅샷 + 이력 로그 이중 저장
  - 장점: 목록 조회 빠름, 감사 추적 쉬움
  - 단점: 쓰기 시 동기화 책임 증가
- 엄격한 상태 전이 제한
  - 장점: 데이터 품질 향상, 통계 왜곡 감소
  - 단점: 예외 처리(재오픈 등) 유연성 낮음
- 평균 처리시간만 제공
  - 장점: 구현 단순, 메시지 명확
  - 단점: 중앙값/백분위수 등 분포 정보 부족

## 7) 포트폴리오 관점에서 강조할 수 있는 인사이트
- "기능 개수"보다 "업무 흐름 완결성"이 내부 시스템 설계의 핵심
- 상태 변경 로그를 중심으로 설계하면 기능 확장(알림/SLA/감사) 기반이 마련됨
- MVP 단계에서도 통계 지표를 포함하면 운영 개선 사이클(측정 -> 개선)의 출발점을 만들 수 있음
