# 00. 범위 정의 (Scope)

## 1) 프로젝트 목적
- 프로젝트명: 민원/요청 처리 통합 관리 시스템(Helpdesk)
- 성격: 외부 고객용 서비스가 아닌, 사내 운영/지원 부서가 사용하는 내부 업무 시스템
- 포트폴리오 목표: "복잡도를 통제한 MVP를 빠르게 구현하고, 운영 흐름을 데이터로 추적하는 능력"을 보여주는 것

## 2) 문제 정의
- 현재(가정) 내부 요청이 메신저/이메일/구두로 분산되어 누락, 중복, 지연이 발생
- 요청 처리 상태가 담당자 개인에게 종속되어 팀 단위 가시성이 낮음
- 처리 이력이 남지 않아 사후 회고(원인 분석, 병목 파악)가 어려움
- 평균 처리시간 같은 기본 운영 지표가 없어 개선 우선순위 설정이 어려움

## 3) 사용자와 역할 (MVP 단순화)
- 요청자(직원)
  - 티켓 등록
  - 본인이 등록한 티켓 목록/상태 확인
- 처리자(헬프데스크 담당)
  - 티켓 상태 변경(접수/처리중/완료)
  - 처리 메모(이력 로그) 기록
- 공통
  - 기본 통계 확인(평균 처리시간)

> 주의: MVP에서는 복잡한 권한 체계(RBAC, 부서별 세분화)는 제외하고, 역할을 최소 수준으로만 구분

## 4) MVP 포함 범위 (In Scope)
- 티켓 등록
  - 제목, 내용, 요청자, 카테고리(단순 분류), 생성시각
- 상태 변경
  - 상태: `접수` -> `처리중` -> `완료`
  - MVP에서는 역전이/복잡 전이는 미지원(단방향 중심)
- 처리 이력 로그
  - 누가/언제/어떤 상태로 바꿨는지 + 메모
- 간단 통계
  - 완료된 티켓 기준 평균 처리시간(생성~완료)

## 5) MVP 제외 범위 (Out of Scope)
- 알림(메일/슬랙/사내 메신저)
- 자동 분류 AI/우선순위 추천
- 복잡한 권한 체계(조직도 연동, 세밀한 접근제어)
- 외부 시스템 연동(SSO, 그룹웨어, ITSM, ERP 등)

## 6) 성공 기준 (MVP 완료 정의)
- 내부 요청이 최소한 "티켓 단위"로 누락 없이 등록된다.
- 각 티켓의 현재 상태와 변경 이력을 UI에서 확인할 수 있다.
- 완료 티켓의 평균 처리시간을 집계해 운영 지표로 볼 수 있다.
- 데모 시나리오(등록 -> 처리중 -> 완료 -> 통계 반영)가 끊김 없이 재현된다.

## 7) 비기능 요구사항 (MVP 수준)
- 사용성: 복잡한 설정 없이 바로 사용할 수 있는 단순 화면 흐름
- 추적성: 상태 변경은 반드시 이력 로그로 남아야 함
- 일관성: 티켓 상태와 이력 로그 간 불일치가 없어야 함(트랜잭션 처리)
- 확장성(후속 단계 대비): 상태/카테고리/통계를 확장할 수 있는 구조

## 8) 리스크 및 대응
- 리스크: 범위 확장 유혹(알림, 권한, SLA, 대시보드 고도화)
  - 대응: 본 문서의 In/Out Scope를 기준으로 스프린트 단위 범위 통제
- 리스크: 내부 시스템 느낌 약화(과도한 고객지원형 UX)
  - 대응: 용어/화면을 "요청자-처리자-운영지표" 중심으로 설계
- 리스크: 통계 신뢰도 저하(완료시각 누락, 상태 전이 예외)
  - 대응: 완료 전이 시점에 완료시각 강제 기록, 통계 쿼리 기준 명확화

## 9) 단계별 확장 후보 (MVP 이후)
- 알림 훅(이메일/메신저)
- 필터/검색 고도화(담당자, 기간, 카테고리)
- SLA 지표(응답시간, 처리시간 분리)
- 권한 모델 정교화(팀장/관리자/감사뷰어)
