# 03. 면접 시연 시나리오 (1분/3분)

## 목적
- 짧은 시간 안에 "내부 업무 시스템으로서의 완결성"을 보여준다.
- 핵심 메시지: 추적 가능성, 상태 일관성, 이력 관리, 설명 가능성.

## 시연 전 준비
1. 백엔드 실행
```bash
cd C:\Users\qjawn\helpdesk-system\backend
npm run dev
```

2. 프론트 실행
```bash
cd C:\Users\qjawn\helpdesk-system\frontend
npm run dev
```

3. 접속
- 프론트: `http://localhost:5173`
- 백엔드: `http://localhost:3000`

## 1분 버전 (핵심만)
1. `ADMIN`으로 티켓 1건 등록
2. `RECEIVED -> IN_PROGRESS -> DONE` 상태 변경
3. 상세에서 이력(`CREATED`, `STATUS_CHANGED`) 확인
4. 통계(완료건수/평균/중앙값) 확인
5. `REQUESTER`로 전환해 상태변경 시 `403` 확인

한 줄 설명:
- "기능을 넓히지 않고, 내부 운영에 중요한 추적/일관성/권한을 최소 단위로 검증했습니다."

## 3분 버전 (면접용 표준)

### 1) ADMIN 시연 (약 2분)
1. 역할/사용자 설정
- 역할: `ADMIN`, 사용자: `admin1`

2. 티켓 등록
- 요청자/제목/내용 입력 후 등록
- 설명 포인트: "요청 접수를 티켓 단위로 표준화"

3. 목록/검색 확인
- 상태 필터 적용
- 요청자 필터, 키워드 검색 적용
- 설명 포인트: "운영자가 필요한 건을 빠르게 찾는 최소 검색 기능"

4. 상태 변경
- `RECEIVED -> IN_PROGRESS -> DONE`
- 설명 포인트: "상태 전이 규칙으로 데이터 일관성 보장"

5. 상세/이력 확인
- 시간순 로그 확인 (`CREATED`, `STATUS_CHANGED`)
- 설명 포인트: "누가/언제/무엇을 변경했는지 추적 가능"

6. 통계 확인
- 평균 처리시간, 중앙값, 일자별 완료 건수 확인
- 설명 포인트: "운영 개선의 출발점이 되는 최소 KPI"

### 2) REQUESTER 시연 (약 1분)
1. 역할/사용자 설정
- 역할: `REQUESTER`, 사용자: `user1`

2. 접근 제어 확인
- 목록에서 본인 티켓만 보임
- 상태 변경 시도 시 `403 Forbidden`
- 설명 포인트: "복잡 RBAC 없이도 최소 권한 분리 구현"

## 예상 질문 답변 포인트
- Q. 왜 단순하게 만들었나?
  - A. 면접 과제 목적상 기능 수보다 핵심 업무 흐름의 완결성과 설명 가능성을 우선했다.
- Q. 왜 상태를 3단계로 제한했나?
  - A. 전이 규칙을 명확히 해 데이터 일관성과 테스트 가능성을 높였다.
- Q. 왜 이 통계만 넣었나?
  - A. 평균/중앙값/일자별 완료만으로도 운영 개선 대화가 가능하고, 범위 통제를 유지할 수 있다.
- Q. 권한이 너무 단순한 것 아닌가?
  - A. 맞다. 복잡 권한은 제외하고, MVP+ 단계에서 최소 역할 분리만 도입했다.

## 최종 체크리스트
- [ ] ADMIN 플로우(등록 -> 상태변경 -> 상세 -> 통계)가 끊김 없이 동작
- [ ] REQUESTER 권한 제한(조회 범위/상태변경 차단)이 동작
- [ ] 설명에서 범위 외 기능(알림/AI/복잡 권한/외부 연동)을 확장 약속하지 않음
